# 多方块机械父类开发说明

## 目的

现在多方块机械的通用逻辑已经下沉到 `MachineControllerBlockEntity`。

相关类型现在统一放在：

```java
dev.celestiacraft.cmi.api.register.multiblock.machine
```

`multiblock` 保留多方块控制器本身的通用内容，`multiblock.machine` 专门放多方块机械相关内容。

这层父类主要负责三件事：

- 统一管理 `Item / Fluid / Energy` capability
- 统一处理内部库存、流体、能量的 NBT 存取
- 统一校验多方块结构里的 IO 数量和 IO 模式

这样做的目的很简单：

- 子类只负责声明机器配置和配方逻辑
- 少写重复代码
- 避免每台机器各写一套能力包装，后面维护越来越乱

---

## 现在应该怎么写

普通多方块机械优先继承：

```java
MachineControllerBlockEntity
```

如果机器本身有配方 tick 逻辑，再实现：

```java
IControllerRecipe<当前方块实体类>
```

例如：

```java
public class ExampleBlockEntity extends MachineControllerBlockEntity
		implements IControllerRecipe<ExampleBlockEntity> {
}
```

如果实现了 `IControllerRecipe`，还需要提供一个稳定的配方类型 ID：

```java
@Override
public ResourceLocation getRecipeTypeId() {
	return Cmi.loadResource("example_machine");
}
```

这个 ID 可以先用于显示和记录。

如果后面项目里真的注册了同名 `RecipeType`，接口默认也能把它解析出来。

推荐做法是在 `CmiRecipeType` 里用统一 helper 注册同名类型：

```java
public static final Supplier<RecipeType<?>> EXAMPLE_MACHINE;

static {
	EXAMPLE_MACHINE = registerSimple("example_machine");
}
```

这样接口里返回的：

```java
Cmi.loadResource("example_machine")
```

就会和注册表里的类型保持一致。

---

## 父类已经接管的内容

下面这些内容，通常不需要子类自己再写：

- `getCapability`
- `onLoad`
- `invalidateCaps`
- `reviveCaps`
- `getUpdateTag`
- `getUpdatePacket`
- 内部物品槽序列化
- 内部流体槽序列化
- 内部能量序列化
- 结构有效性 + IO 数量校验
- `Item / Fluid / Energy` 的基础 capability 包装

如果子类重新写这些方法，先确认确实有必要。

---

## 子类主要负责什么

### 1. 声明机器的基础配置

常用的是下面几个方法：

```java
@Override
protected int getItemSlots() {
	return 8;
}

@Override
protected FluidSlots[] getFluidSlots() {
	return new FluidSlots[] {
			new FluidSlots(1000, IOMode.INPUT),
			new FluidSlots(4000, IOMode.OUTPUT)
	};
}

@Override
protected int getEnergyCapacity() {
	return 10000;
}
```

含义分别是：

- `getItemSlots()`：内部物品槽数量
- `getFluidSlots()`：内部流体槽定义，包含容量和模式
- `getEnergyCapacity()`：内部总能量容量

---

### 2. 声明结构里允许多少个 IO 点

这些方法控制的是“结构合法性”，不是容量。

例如：

```java
@Override
protected int getMinFluidIO() {
	return 1;
}

@Override
protected int getMaxFluidIO() {
	return 1;
}
```

这表示：

- 结构里最少要有 1 个流体 IO
- 最多也只能有 1 个流体 IO

合起来就是：必须正好有 1 个。

可用方法有：

- `getMinItemIO()`
- `getMaxItemIO()`
- `getMinFluidIO()`
- `getMaxFluidIO()`
- `getMinEnergyIO()`
- `getMaxEnergyIO()`

如果数量不在范围内，父类会把机器视为无效，能力也不会正常暴露。

---

### 3. 声明输入输出模式

#### Item

```java
@Override
protected IOMode getItemIO(int slot) {
	return IOMode.INPUT;
}
```

#### Fluid

一般不需要额外重写 `getFluidIO(int slot)`。

父类默认会直接读取 `getFluidSlots()` 里每个槽位的 `mode`。

只有在流体模式不是固定写在 `FluidSlots` 里时，才考虑重写。

#### Energy

```java
@Override
protected IOMode getEnergyIO(int slot) {
	return IOMode.BOTH;
}
```

---

## 什么情况下不用重写

下面这些情况，建议直接交给父类默认实现：

### `getItemIO(int slot)`

父类默认就是：

```java
IOMode.BOTH
```

如果你的机器物品槽本来就是双向，就不要再写一层：

```java
return super.getItemIO(slot);
```

也不要重复写：

```java
return IOMode.BOTH;
```

直接删掉这个 override 就行。

### `getEnergyIO(int slot)`

同上。默认也是 `IOMode.BOTH`。

### `getFluidIO(int slot)`

如果流体模式已经写在 `getFluidSlots()` 里，就不要重复写。

例如：

```java
@Override
protected FluidSlots[] getFluidSlots() {
	return new FluidSlots[] {
			new FluidSlots(4000, IOMode.OUTPUT)
	};
}
```

这种情况下，不需要再额外写 `getFluidIO()`。

---

## 外部 IO 方块的机器怎么写

有些机器不是控制器自己存库存，而是通过结构里的独立 IO 方块工作。

这种机器可以参考 `TestCokeOvenBlockEntity`。

要点是：

- 关闭控制器自己的对应 capability
- 关闭控制器自己的内部存储
- 通过结构查找 IO 方块实体
- 配方逻辑直接操作 IO 方块

例如：

```java
@Override
protected boolean supportsControllerItemIO() {
	return false;
}

@Override
protected boolean useInternalItemStorage() {
	return false;
}
```

结构里的 IO 数量统计可以自己重写：

```java
@Override
protected int getActualItemIOCount() {
	return countMatchedBlocks(CmiBlock.TEST_COKE_OVEN_IO.get());
}
```

这样父类仍然能负责“结构里有几个 IO 点才算合法”的校验。

## `IControllerRecipe` 的写法

现在 `IControllerRecipe` 已经带泛型，不要再用原始类型。

正确写法：

```java
public class ExampleBlockEntity extends MachineControllerBlockEntity
		implements IControllerRecipe<ExampleBlockEntity> {

	@Override
	public MultiblockContext<ExampleBlockEntity> tick(MultiblockContext<ExampleBlockEntity> context) {
		return context;
	}

	@Override
	public void recipe(MultiblockContext<ExampleBlockEntity> context) {
	}
}
```

这样 IDE 能正确推断：

- `context.getEntity()` 的具体类型
- `tick` / `recipe` 的上下文类型

不容易再出现原始类型 warning。

另外，`IControllerRecipe` 现在还要求实现：

```java
ResourceLocation getRecipeTypeId()
```

并提供了默认方法：

```java
RecipeType<?> getRecipeType()
```

它会尝试通过 `ForgeRegistries.RECIPE_TYPES` 按 ID 解析真正的 `RecipeType`。

这意味着：

- 你现在就可以记录和展示这个 ID
- 以后注册了实际的 `RecipeType` 以后，可以直接复用这套接口
- 推荐在 `CmiRecipeType` 里直接注册同名类型，减少额外映射

如果某个 ID 目前还没有对应的注册类型，`getRecipeType()` 会返回 `null`，这是正常情况。

## 特殊工作条件的写法

如果机器除了结构和 IO 之外，还需要满足额外条件才能运行，例如：

- 必须在白天
- 必须在某个维度
- 必须在指定群系
- 必须露天
- 必须下雨
- 周围必须有辅助方块或热源

不要把这类判断直接散落在每个 `recipe()` 里。

现在父类已经提供了统一入口：

```java
protected WorkConditionResult getWorkCondition(MultiblockContext<? extends MachineControllerBlockEntity> context) {
	return WorkConditionResult.success();
}
```

默认返回通过。

如果子类有额外要求，就重写这个方法。

例如，要求机器只能在白天工作：

```java
@Override
protected WorkConditionResult getWorkCondition(MultiblockContext<? extends MachineControllerBlockEntity> context) {
	if (!context.getLevel().isDay()) {
		return WorkConditionResult.fail("need_daytime");
	}

	return WorkConditionResult.success();
}
```

这里的 `reasonKey` 只是一个失败原因标识。

它可以用于：

- tooltip
- Jade / Waila 提示
- 调试信息
- 后续本地化文本映射

### 在 tick 里怎么用

推荐写法：

```java
@Override
public MultiblockContext<ExampleBlockEntity> tick(MultiblockContext<ExampleBlockEntity> context) {
	if (prepareRecipeTick(context)) {
		recipe(context);
	}
	return context;
}
```

`prepareRecipeTick(context)` 已经统一做了这些事：

- 客户端直接跳过
- 结构无效直接跳过
- IO 数量不合法直接跳过
- 工作条件不满足时直接跳过

所以正常情况下，`tick()` 里优先用它，不要自己再重复写一遍。

### 如果条件失败时要重置进度

父类还提供了这个方法：

```java
protected void onWorkConditionFailed(
		MultiblockContext<? extends MachineControllerBlockEntity> context,
		WorkConditionResult result
) {
}
```

默认不做事。

如果某台机器在条件不满足时要清空进度、熄火、停止动画，可以重写这里。

例如：

```java
@Override
protected void onWorkConditionFailed(
		MultiblockContext<? extends MachineControllerBlockEntity> context,
		WorkConditionResult result
) {
	workTimer = 0;
}
```

### 什么时候用它，什么时候不用

适合放进 `getWorkCondition()` 的内容：

- 环境限制
- 世界状态限制
- 结构外部辅助条件
- 与配方执行前置相关，但不属于库存/槽位校验的条件

不适合放进 `getWorkCondition()` 的内容：

- 输入物品够不够
- 输出槽能不能放下
- 流体够不够
- 能量够不够

这些仍然属于具体配方逻辑，应该留在 `recipe()` 里。

## `FluidFilters` 的使用

现在流体标签过滤不要再走 `builtInRegistryHolder()`。

已经改成通过 Forge 的 tag 管理器判断：

```java
ForgeRegistries.FLUIDS.tags().getTag(tag).contains(stack.getFluid())
```

可直接这样写：

```java
new FluidSlots(1000, IOMode.INPUT, FluidFilters.tag(FluidTags.WATER))
```

---

## 旧的 TestMultiblock 四个 capability 类现在对应到哪里

原来这几个类：

- `TestEnergyCapability`
- `TestFluidCapability`
- `TestItemCapability`
- `TestItemHandler`

已经不再单独保留。

现在的对应关系：

- `TestItemCapability` -> 父类内部的 `createInternalItemStorage()`
- `TestItemHandler` -> 父类内部的 `MachineItemHandler`
- `TestFluidCapability` -> 父类内部的 `MachineFluidHandler`
- `TestEnergyCapability` -> 父类内部的 `MachineEnergyStorage`

`TestMultiblockBlockEntity` 现在只需要声明：

- IO 数量范围
- 槽位数量
- 流体槽配置
- 能量容量

剩下的由父类统一处理。

---

## 开发时的几个约定

- 只要父类默认行为已经满足需求，就不要保留空的 override
- `super.xxx()` 纯委托一般没有必要，直接删掉更清楚
- `FluidSlots` 已经能表达流体模式时，不要再额外写一遍 `getFluidIO()`
- 结构里使用外部 IO 方块时，记得区分“控制器自身 capability”和“结构中的实际 IO 数量”
- 新机器优先先把配置方法写清楚，再写配方逻辑，不要一开始就把逻辑塞进 `getCapability`

---

## 最后

这套父类不是为了把所有机器都写死成一种样子，而是把重复、琐碎、容易出错的底层逻辑收口。

子类应该尽量保持简单：

- 机器有什么
- 结构要求什么
- 配方怎么跑

只写这三件事，基本就够了。

---

## KubeJS 固定方法写法

现在多方块机器的 KubeJS 兼容，开始支持参考 MBD 那种固定方法风格。

也就是说，后面不再优先让脚本作者去记一堆 JSON key，
而是直接调用现成方法。

目前已经接通的例子是：

```java
cmi:test_coke_oven
```

可直接使用的方法有：

- `duration(...)`
- `inputItem(...)`
- `outputItem(...)`
- `inputFluid(...)`
- `outputFluid(...)`
- `inputEnergy(...)`
- `outputEnergy(...)`

同时保留了复数别名：

- `inputItems(...)`
- `outputItems(...)`
- `inputFluids(...)`
- `outputFluids(...)`

### 示例

```js
ServerEvents.recipes(event => {
  event.recipes.cmi.test_coke_oven()
    .inputItem('#minecraft:logs')
    .outputItem(Item.of('minecraft:charcoal', 1))
    .outputFluid(Fluid.of('immersiveengineering:creosote', 125))
    .duration(20)
    .id('cmi:kjs/test_coke_oven/charcoal')
})
```

当前先接通的是 `test_coke_oven` 这条链路，目的是先把整套模式跑通。

也就是：

- 注册了 `cmi:test_coke_oven` 的 recipe type
- 注册了同名 serializer
- KubeJS schema 改成固定方法 builder
- `TestCokeOvenBlockEntity` 会真的按这个 recipe type 读配方执行

### 后续扩展时怎么做

如果别的多方块机器也要复用这套 KubeJS 固定方法模式，通常只需要三步：

1. 在机器 BlockEntity 里实现 `getRecipeTypeId()`
2. 在 `CmiRecipeType` 里注册同名 recipe type
3. 在 `CmiRecipeSerializer` 和 `ModKubeJSPlugin` 里注册对应 serializer / schema

如果配方结构还是同一套 `item / fluid / energy / duration` 格式，直接复用这一套 schema 就够了。
